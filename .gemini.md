# Security Remediation Report

## Identified Issues

Based on the Supabase database linter, the following security vulnerabilities were detected:

1. **RLS Disabled in Public (`rls_disabled_in_public`)**
   Row Level Security (RLS) has not been enabled on tables exposed to PostgREST in the `public` schema. This allows any API user (authenticated or anonymous) to potentially query or manipulate all records in these tables.
   - Affected Tables: `public.trips`, `public.users`, `public.trip_days`, `public.trip_places`, `public.place_cache`, `public.user_profiles`, `public.travel_issues`, `public.refresh_tokens`.

2. **Sensitive Columns Exposed (`sensitive_columns_exposed`)**
   The `public.users` table contains a potentially sensitive column (`password`) which is exposed via the API without RLS protection, leading to potential data exposure.

## Resolution Actions

A database migration script (`supabase/migrations/20260312000001_security_fixes.sql`) was created to implement the following mitigations:

1. **Enable Row Level Security (RLS)**
   RLS has been explicitly enabled for all affected tables in the `public` schema. By default, enabling RLS sets up a "default deny" rule, blocking all operations (SELECT, INSERT, UPDATE, DELETE) via the PostgREST API unless an explicit policy grants access.
   ```sql
   ALTER TABLE IF EXISTS public.<table_name> ENABLE ROW LEVEL SECURITY;
   ```

2. **Protect Sensitive Columns**
   To prevent the exposure of the `password` column in the `public.users` table, a script was added to revoke SELECT permissions for the `anon` and `authenticated` roles specifically for this column.
   ```sql
   REVOKE SELECT (password) ON public.users FROM public, anon, authenticated;
   ```

## Status
All identified vulnerabilities have been addressed in the migration file. The changes enforce secure default access controls (RLS) and hide sensitive information from public-facing API endpoints.

## ⚠️ Action Required: Manual Execution in Supabase SQL Editor

The generated migration queries **must be manually executed** in your Supabase project's SQL Editor to take effect. 

The security fixes have been automatically appended to the very end of `supabase/combined_migrations.sql` (`[11/11] 20260312000001_security_fixes.sql`). 

Please open the Supabase Dashboard, navigate to the **SQL Editor**, and execute the following query:

```sql
-- Fix RLS Disabled in Public issues
ALTER TABLE IF EXISTS public.trips ENABLE ROW LEVEL SECURITY;
ALTER TABLE IF EXISTS public.users ENABLE ROW LEVEL SECURITY;
ALTER TABLE IF EXISTS public.trip_days ENABLE ROW LEVEL SECURITY;
ALTER TABLE IF EXISTS public.trip_places ENABLE ROW LEVEL SECURITY;
ALTER TABLE IF EXISTS public.place_cache ENABLE ROW LEVEL SECURITY;
ALTER TABLE IF EXISTS public.user_profiles ENABLE ROW LEVEL SECURITY;
ALTER TABLE IF EXISTS public.travel_issues ENABLE ROW LEVEL SECURITY;
ALTER TABLE IF EXISTS public.refresh_tokens ENABLE ROW LEVEL SECURITY;

-- Fix Sensitive Columns Exposed
DO $$
BEGIN
  IF EXISTS (
      SELECT 1 
      FROM information_schema.columns 
      WHERE table_schema = 'public' 
        AND table_name = 'users' 
        AND column_name = 'password'
  ) THEN
    REVOKE SELECT (password) ON public.users FROM public, anon, authenticated;
  END IF;
END $$;
```
